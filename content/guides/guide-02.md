"프롬프트를 잘 써야 한다"는 말은 많이 들었을 것이다.  
하지만 "어떻게 잘 써야 하는지"를 구체적으로 알려주는 곳은 드물다.

"자세하게 써라", "명확하게 써라" — 이런 말은 막연하다. 실제로 결과가 달라지는 건 **구체적인 기법**을 알 때부터다.

이 글에서는 프롬프트 엔지니어링의 핵심 기법 8가지를 소개한다. 각 기법마다 잘못된 예시와 올바른 예시를 나란히 두고, 왜 다른지를 분석한다.

---

## 기법 1: 역할 부여 (Role Assignment)

### 원리

AI는 "누구로서 답하느냐"에 따라 어휘, 구조, 깊이가 모두 달라진다. 단순히 "전문가처럼 써줘"와 세밀한 페르소나를 지정하는 것의 차이는 생각보다 크다.

### 예시 비교

❌ **잘못된 예시**
```
전문가처럼 마케팅 전략을 짜줘.
```

✅ **올바른 예시**
```
너는 B2B SaaS 스타트업에서 10년간 성장 마케팅을 이끌어온 CMO다.
지금 막 시리즈 A를 받은 HR 자동화 솔루션 회사의 초기 6개월 GTM(시장 진출) 전략을 짜줘.
타겟은 50~200인 규모의 스타트업 HR 담당자다.
예산은 월 500만 원이고, 현재 팀은 마케터 1명뿐이다.
```

### 왜 다른가

첫 번째 프롬프트는 AI가 "마케팅 전문가"를 어떻게 정의할지 스스로 결정한다. 그 결과는 교과서적이고 범용적인 조언이 된다. 두 번째는 분야(B2B SaaS), 경력(10년), 직위(CMO), 현재 상황(시리즈 A), 제약(예산, 팀 규모)을 모두 명시했다. AI는 이 맥락 안에서만 생각하므로, 훨씬 구체적이고 실행 가능한 전략을 내놓는다.

**페르소나 지정 시 포함할 요소:**
- 직책 + 경력 연수
- 전문 분야 (어떤 산업, 어떤 규모)
- 현재 상황의 제약 조건

## 기법 2: 출력 형식 강제 (Output Format Control)

### 원리

AI에게 "잘 써줘"라고 하면 AI가 형식을 스스로 결정한다. 결과물의 형태를 명시하면 후처리 작업이 줄고, 원하는 구조를 바로 얻을 수 있다.

### 예시 비교

❌ **잘못된 예시**
```
우리 서비스의 장점 3가지를 정리해줘.
```

✅ **올바른 예시**
```
우리 서비스의 장점 3가지를 아래 형식으로 정리해줘:

| 장점 | 한 줄 설명 | 고객이 얻는 혜택 |
|---|---|---|

각 행은 20자 이내로 작성하고, 전문 용어 없이 일반인이 이해할 수 있는 언어를 써줘.
```

### 왜 다른가

형식을 지정하지 않으면 AI는 글쓰기 스타일을 자유롭게 선택한다. 경우에 따라서는 긴 문단으로 쓰거나, 불필요한 서론을 붙이거나, 원하는 구조와 다른 방식으로 정리한다. 표 형식을 명시하면 추가 편집 없이 바로 사용할 수 있는 결과물이 나온다.

**자주 쓰는 형식 지정:**
- 표 (| 컬럼 | 컬럼 |)
- 번호 목록 (1, 2, 3)
- JSON (개발 연동 시)
- 마크다운 헤딩 구조 (H2, H3)
- 글자 수 제한 ("각 항목은 50자 이내로")

## 기법 3: Few-Shot 예시 주입

### 원리

AI에게 원하는 결과의 스타일을 "설명"하는 것보다 **예시를 보여주는 것**이 훨씬 효과적이다. 예시 1~3개만 넣어도 결과물의 톤과 형식이 극적으로 달라진다.

### 예시 비교

❌ **잘못된 예시**
```
짧고 임팩트 있는 마케팅 카피를 써줘. 감정을 자극해야 하고 너무 공격적이면 안 돼.
```

✅ **올바른 예시**
```
아래 스타일과 비슷한 마케팅 카피를 5개 써줘.

예시 1: "당신의 첫 번째 투자, 나중에 시작하면 이미 늦습니다."
예시 2: "매일 30분. 1년 후 당신의 영어가 달라집니다."
예시 3: "복잡한 세금 신고, 이제 15분이면 됩니다."

위 예시들의 공통점: 숫자 포함, 한 문장 혹은 두 문장, 독자의 변화에 초점, 위협감 없이 동기 부여.

작성할 제품: 중소기업을 위한 AI 회계 솔루션
```

### 왜 다른가

"짧고 임팩트 있게"는 주관적인 기준이다. AI마다, 실행마다 다르게 해석한다. 예시를 보여주면 AI는 그 패턴을 분석하고 복제한다. 특히 톤, 문장 구조, 길이, 사용 언어의 수준 같은 미묘한 요소들이 예시 없이는 전달하기 매우 어렵다.

## 기법 4: 체인 오브 소트 (Chain of Thought)

### 원리

복잡한 문제를 풀 때 AI에게 "답을 내놓아라"고 바로 요청하는 것보다 "단계별로 생각해라"고 지시하면 정확도가 올라간다. AI가 중간 추론 과정을 명시적으로 거치기 때문이다.

### 예시 비교

❌ **잘못된 예시**
```
이 계약서에서 우리 회사에 불리한 조항이 있는지 알려줘.
[계약서 내용]
```

✅ **올바른 예시**
```
이 계약서를 분석해줘. 다음 순서로 단계별로 생각해:

1단계: 각 조항을 하나씩 읽고 무엇을 규정하는지 요약해
2단계: 각 조항이 계약 당사자 중 어느 쪽에 유리한지 판단해
3단계: 우리 회사(을)의 입장에서 잠재적 리스크가 있는 조항을 표시해
4단계: 리스크 수준을 높음/중간/낮음으로 분류하고 이유를 설명해
5단계: 수정을 제안하거나 삭제를 고려해야 할 조항을 최종 정리해

[계약서 내용]
```

### 왜 다른가

AI는 즉각적인 답변을 요구받으면 종종 중간 논리를 건너뛰고 결론으로 점프한다. 이 과정에서 오류가 생긴다. 단계를 명시하면 AI가 각 단계를 실제로 처리하고, 그 결과가 다음 단계의 입력이 된다. 수학 문제, 논리 분석, 다단계 의사결정 등 복잡한 작업에서 특히 효과가 크다.

**Chain of Thought가 효과적인 상황:**
- 복잡한 문서 분석
- 논리적 판단이 필요한 상황
- 수학적 계산
- 여러 요소를 동시에 고려해야 하는 의사결정

## 기법 5: 시스템 프롬프트 vs 유저 프롬프트 분리 전략

### 원리

대부분의 AI API와 고급 도구는 **시스템 프롬프트**와 **유저 프롬프트**를 별도로 설정할 수 있다. 시스템 프롬프트는 AI의 전반적인 행동 방식을 정의하고, 유저 프롬프트는 개별 요청을 담는다.

### 예시 비교

❌ **잘못된 예시 (매번 같은 지시를 반복)**
```
유저: 너는 고객 서비스 전문가야. 항상 공손하게 답변하고, 불확실한 내용은 "확인이 필요합니다"라고 말해. 
그리고 우리 제품은 구독 기반이고 환불은 7일 이내만 가능해. 고객: 제품이 마음에 안 들어요.

유저: 너는 고객 서비스 전문가야. 항상 공손하게 답변하고... [반복]
고객: 가격을 낮춰줄 수 있나요?
```

✅ **올바른 예시 (분리 구조)**
```
[시스템 프롬프트 — 한 번만 설정]
너는 [회사명]의 고객 서비스 담당자다.
- 항상 공손하고 간결하게 답변한다
- 불확실한 내용은 "확인이 필요합니다. 담당자 연결을 도와드리겠습니다"로 응답한다
- 환불 정책: 구매 후 7일 이내만 가능
- 구독 요금제: 월 29,000원 / 연 290,000원
- 불가능한 요청(할인, 정책 예외 등)은 정중하게 거절하되 대안을 제시한다

[유저 프롬프트 — 매 요청마다]
고객: 제품이 마음에 안 들어요.
```

### 왜 다른가

시스템 프롬프트에 규칙과 컨텍스트를 담으면 매 대화마다 반복할 필요가 없다. 특히 실제 서비스를 구축할 때, 시스템 프롬프트에 회사 정책, 답변 톤, 금지 사항을 고정해두면 일관성 있는 결과물을 얻는다. API 비용도 절감된다.

## 기법 6: 제약 조건 명시 (Constraints)

### 원리

AI에게 "하지 말아야 할 것"을 명시하는 것은 "해야 할 것"을 명시하는 것만큼 중요하다. 금지 조건을 넣으면 AI가 자주 하는 실수 패턴을 방지할 수 있다.

### 예시 비교

❌ **잘못된 예시**
```
스타트업 창업자를 위한 블로그 글을 써줘. 주제는 팀 빌딩이야.
```

✅ **올바른 예시**
```
스타트업 창업자를 위한 블로그 글을 써줘. 주제는 팀 빌딩이야.

하지 말 것:
- "중요합니다", "필수입니다" 같은 당연한 말 반복 금지
- 이미 잘 알려진 내용("소통이 중요하다" 등) 나열 금지
- 서론에서 주제를 2번 이상 예고하지 말 것
- 마지막에 "이상으로 살펴보았습니다" 같은 공허한 결론 금지
- 영어 원문 용어를 그대로 나열하지 말 것 (반드시 한국어로 설명)

해야 할 것:
- 실제 실패 사례 또는 구체적 수치가 포함된 예시 1개 이상
- 독자가 내일 당장 실행 가능한 행동 지침으로 끝낼 것
```

### 왜 다른가

AI는 학습 데이터에서 자주 등장하는 패턴을 따르는 경향이 있다. 블로그 글이라면 "서론-본론-결론", "중요합니다-필수입니다" 같은 공식적이고 무난한 패턴이 확률적으로 높다. 금지 조건을 명시하면 AI가 이런 디폴트 패턴을 피하고 더 구체적이고 차별화된 결과물을 낸다.

## 기법 7: 반복 정제 (Iterative Refinement)

### 원리

한 번의 프롬프트로 완성본을 얻으려는 시도는 대부분 실패한다. 프롬프트를 단계적으로 쌓아가며 결과물을 개선하는 접근이 훨씬 효과적이다.

### 예시: 3단계 정제 흐름

**1단계 — 구조 먼저**
```
다음 주제로 LinkedIn 포스트를 쓸 거야: "AI 도구 도입 후 실제로 업무가 얼마나 줄었는가"

지금은 글 전체를 쓰지 말고, 구조(목차)만 3가지 안을 제시해줘.
각 안의 핵심 차별점을 한 줄로 설명해줘.
```

**2단계 — 초안 생성**
```
2번 안으로 초안을 써줘. 
- 700자 내외
- 첫 줄은 질문이나 놀라운 수치로 시작
- 나의 실제 경험담 형식 (1인칭)
- 마지막은 독자에게 질문으로 마무리
```

**3단계 — 편집 지시**
```
위 글에서:
1. 3번째 문단이 너무 일반적임. 구체적 수치나 예시로 교체해줘
2. 전반적으로 문장이 긴 편임. 3줄 이상인 문단은 나눠줘
3. "중요합니다"로 끝나는 문장을 모두 삭제하거나 교체해줘
```

### 왜 다른가

한 번에 완성하려 하면 AI도 압축적으로 판단을 내린다. 단계를 나누면 각 단계에서 집중도가 높아지고, 중간에 방향을 조정할 기회도 생긴다. 또한 AI의 현재 결과물을 보고 다음 지시를 조정할 수 있어 최종 결과물의 품질이 훨씬 높아진다.

## 기법 8: 메타 프롬프트 — AI에게 프롬프트 자체를 개선시키기

### 원리

좋은 프롬프트를 스스로 작성하는 능력이 부족할 때, AI에게 프롬프트를 대신 만들어달라고 요청할 수 있다. 이것이 **메타 프롬프트**다.

### 예시 비교

❌ **잘못된 예시 (막막한 상황)**
```
좋은 프롬프트를 써줘.
```

✅ **올바른 예시 (메타 프롬프트)**
```
나는 AI에게 다음 작업을 시키고 싶어:
"고객 인터뷰를 요약해서 제품팀에 전달할 인사이트 리포트 만들기"

이 작업을 위한 최적의 프롬프트를 작성해줘.
프롬프트에는 다음이 포함돼야 해:
- 역할 지정
- 입력 형식 명시
- 출력 형식 명시
- 자주 발생하는 실수를 방지하는 제약 조건

작성 후, 이 프롬프트가 효과적인 이유를 간략히 설명해줘.
```

### 응용: 현재 프롬프트 개선 요청

이미 프롬프트를 갖고 있다면 이렇게 활용할 수도 있다:

```
아래는 내가 사용하는 프롬프트야:
[현재 프롬프트]

이 프롬프트의 문제점을 분석하고, 개선된 버전을 제안해줘.
개선 이유도 항목별로 설명해줘.
```

### 왜 다른가

프롬프트 엔지니어링 자체가 아직 배우는 중이라면, AI를 사용해 프롬프트를 개선하는 것이 가장 빠른 학습 방법이기도 하다. AI는 어떤 요소가 빠졌는지, 어떤 표현이 더 효과적인지를 직접 설명해주므로, 단순히 결과물을 얻는 것 이상으로 프롬프트 작성 능력 자체가 향상된다.

## 기법 정리표

| 기법 | 핵심 원리 | 특히 효과적인 상황 |
|---|---|---|
| 역할 부여 | 페르소나가 출력의 깊이를 결정 | 전문적인 글쓰기, 분석 |
| 출력 형식 강제 | 형식을 지정하면 후처리가 줄어듦 | 표, 코드, 구조적 데이터 |
| Few-Shot | 예시가 설명보다 강력함 | 특정 톤·스타일 복제 |
| Chain of Thought | 단계별 추론이 정확도를 높임 | 복잡한 분석, 의사결정 |
| 시스템/유저 분리 | 규칙은 한 번, 요청은 매번 | 반복 작업, 서비스 구축 |
| 제약 조건 명시 | 금지 항목이 디폴트 패턴을 제거 | 차별화된 글쓰기, 오류 방지 |
| 반복 정제 | 단계별 접근이 완성도를 높임 | 긴 콘텐츠, 중요한 문서 |
| 메타 프롬프트 | AI로 AI를 최적화 | 프롬프트 설계 자체가 어려울 때 |
